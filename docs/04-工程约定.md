# 工程约定（新旧大陆）

本项目是六角格、WEGO 同步回合制兵棋。规则复杂且强调确定性。本文件是工程层面的唯一事实来源，对所有代码与协作生效；与设计规则相关处以 `docs/01-战斗机制.md` 为准。

> 本文件工具无关，不依赖任何特定编辑器或 IDE。贡献流程见根目录 [`CONTRIBUTING.md`](../CONTRIBUTING.md)。

## 0. 总则

- **文档即事实来源**：规则数值改动必须同步更新 `docs/`，保持文档与实现一致。
- **约定优先级**：本文件 > 个人偏好。冲突处先改约定再改代码。
- **可配置优先**：能做成数据/配置的就不要写死在逻辑里。

## 1. 目录与架构

```
src/
├── core/    纯规则引擎：确定性、可测试，无渲染/IO/随机源依赖
├── data/    可配置数据：兵种、地形、学说、卡牌、规模档位（纯数据+类型）
├── ui/      渲染与交互：hex 地图、命令下达、结算回放
├── i18n/    文案与本地化资源
└── assets/  美术/音频等二进制资源（走 Git LFS）
```

**依赖方向单向**：`ui → core → (types)`、`data → (types)`、`ui → i18n`。`core` 永不依赖 `ui`、`data`、`assets`。用 lint 规则或 import 边界检查强制此约束。

## 2. 确定性与模拟（WEGO 的生命线）

1. **纯函数式状态机**：`core` 形如 `(state, commands, rng) => nextState`，不原地修改输入 state（不可变更新），便于快照、回放、撤销。
2. **core 必须纯净**：`src/core/**` 不得 import 渲染、DOM、Node IO、计时器或随机源（`Math.random`、`Date.now`）。
3. **快照优先**：进入战斗结算先冻结全场攻/防/韧性；火力比与歼灭判定一律用快照值，战损在阶段末统一扣除。
4. **结算次序无关**：遍历战斗/单位的顺序不得影响最终状态。需要顺序时用文档规定的固定优先级（如撤退优先于推进），并按稳定 id 显式排序，不依赖 Map/Set 迭代顺序。
5. **随机可重放**：掷骰用注入的**带种子 RNG**，绝不用 `Math.random`。相同 `(state, commands, seed)` 必须产出字节级一致的结果——回放、联机、测试的基石。

## 3. 数值规则约束

1. **整除约束**：单位初始攻/防必须能被初始韧性 N 整除；加载兵种数据时校验，非法即报错，不静默取整。
2. **统一修正管线**：火力比移档走一条管线——`基础比 → 各来源移档累加 → ±2 档总封顶`。每个修正带 `source`（`support`/`night`/`card`/`doctrine`），便于按来源精确抵消。
3. **两轴分离**：火力比移档（档轴，±2 封顶）与地形防御 DRM（骰轴，作用于 3D6 读数）分开实现，互不挤占。
4. **标志位集合化**：夜战/特性标志（`nightVisionKeep`、`nightRangeKeep`、`nightAttackKeep`、`nightMoveKeep`、`ignoreZOC`、`nightZocKeep` 等）做成可运行时增删的集合，支持技能卡临时赋予/移除。

## 4. 数据驱动与资源管线

- **配置格式**：结构化数据用 TS 模块（带类型）或带 schema 校验的 JSON。加载时校验，非法数据快速失败（fail fast）。
- **资源命名**：`类别_名称_变体`，全小写 kebab/snake，禁中文与空格（如 `unit_armor-battalion_idle.png`）。
- **二进制资源走 Git LFS**：图片、音频、字体、视频等纳入 `.gitattributes` 的 LFS 跟踪，避免仓库膨胀。
- **资源与代码分离**：`src/assets/` 只放资源，引用通过 manifest/索引，不在逻辑里散落硬编码路径。

## 5. 存档与序列化

- **状态可序列化**：游戏状态为纯数据（无类实例方法、无循环引用），可直接 JSON 序列化。
- **存档带版本号**：每个存档/回放写入 `schemaVersion`；破坏性结构变更时提供**迁移函数**，旧档可升级，不直接弃用。
- **回放 = 初始状态 + 命令序列 + 种子**：优先存命令流而非逐帧状态，体积小且天然受确定性保障。

## 6. 国际化（i18n）

- **不硬编码面向玩家的文案**：UI 文本走 `i18n` key，不在组件里写死中文/英文字符串。
- 设计文档与代码注释用中文；玩家可见文案以 key 管理，默认语言中文，预留英文等扩展。
- 文案 key 命名按域分层（如 `unit.armor-battalion.name`、`card.artillery-prep.desc`）。

## 7. 代码风格

- **TypeScript `strict: true`**，禁用 `any`（确需用 `unknown` 并收窄）；公共 API 显式标注返回类型。
- 固定集合状态用字符串字面量联合（`'Hidden' | 'Spotted' | 'Identified'`）而非数字 enum，利于序列化与可读。
- **ESLint + Prettier** 统一规则与格式，提交前自动跑；不手工争论格式。
- **命名**：标识符英文（camelCase / PascalCase），注释中文并在关键处引用文档章节（如 `// 见 docs/01〈撤退裁定〉`）。领域术语严格对齐设计文档英文名，不自创同义词。
- 文件名 kebab-case；每个 core 子模块以 `index.ts` 为公开出口，内部实现不被跨模块直接 import。

## 8. 性能与内存

- **热路径零意外分配**：结算与渲染循环避免临时对象、闭包分配，减少 GC 抖动。
- 大集合查询用索引/Map，不在循环里做 O(n²) 扫描。
- 性能优化以**测量为准**（profiling），不臆测；优化前后留基准数据。

## 9. 测试

- 用 **Vitest**。新增或修改 core 规则必须配套测试。
- **确定性测试一等公民**：同输入同种子断言结果字节一致；打乱单位/战斗处理顺序断言结果不变。
- **golden/回放测试**：关键战役场景存「初始状态+命令+种子→期望结果」快照，防回归。
- 边界用例：战斗表、火力比取整（向下、偏向防守）、整除衰减、歼灭判定、控制区与无路可退就地歼灭等。
- 提交前 `npm run test`、`npm run lint`、构建均须通过。

## 10. 版本控制与协作

- **分支**：不直接推 `main`；功能走 `feat/*`、修复 `fix/*`、文档 `docs/*`。
- **提交信息**：建议 Conventional Commits（`feat:`/`fix:`/`docs:`/`test:`/`refactor:`），正文说明改了哪条规则/数据及对应文档章节。
- **PR**：小步提交，附改动摘要与测试说明；CI 通过方可合并。
- **大文件用 Git LFS**，禁止把构建产物、`node_modules`、个人 IDE 配置提交入库（见 `.gitignore`）。

## 11. 构建、环境与 CI

- **Node 版本**用 `.nvmrc` / `engines` 锁定，团队一致。
- `.editorconfig` 统一缩进与换行；`.gitattributes` 固定文本换行为 LF，避免跨平台（Windows/macOS/Linux）差异。
- **CI**（如 GitHub Actions）在每个 PR 跑 `install → lint → test → build`，全绿才可合并。

## 12. 文档与决策记录

- 重大架构/规则取舍写 **ADR**（Architecture Decision Record），放 `docs/adr/`，记录背景、选项、决定与后果。
- 设计文档（`docs/0x-*.md`）是规则事实来源；实现与之偏离时以文档为准或先改文档。

## 13. 发布与版本号

- 遵循 **SemVer**（`主.次.修`）。规则/平衡的破坏性变更升主版本并在 `CHANGELOG.md` 记录，便于存档迁移与对局兼容性判断。
